== General & Architectural Requirements

=== Architecture

- *REQ-ARC-01*: The application shall be developed using the Kotlin programming language.
- *REQ-ARC-02*: The application shall follow the Model-View-ViewModel (MVVM) architectural pattern to ensure a clean separation of concerns.
- *REQ-ARC-03*: The user interface shall be implemented using Compose Multiplatform to enable a single codebase for the UI across all supported platforms.

=== Supported Platforms

- *REQ-PLT-01*: The application must be fully functional on the following platforms:
    ** Android (minimum Android 5.0 (API level 21))
    ** iOS (minimum iOS13)
    ** Windows (minimum Windows 10 (x86-64, arm64))
    ** Linux (Ubuntu 20.04 (x86-64, arm64))
    ** macOS (minimum macOS 12 x64, macOS 13 arm64)
    ** Web (Browsers with WasmGC support)

=== Data Storage

- *REQ-STO-01*: All user data, including profiles and health metrics, must be stored exclusively on the user's local device.
- *REQ-STO-02*: The application shall not implement any cloud-based or remote server synchronization for user data. This is a strict privacy requirement.

=== Data Synchronization

- *REQ-SYN-01*: The application shall provide a mechanism for users to synchronize their data between different devices running FitBe.
- *REQ-SYN-02*: This synchronization must occur over a local network (e.g., Wi-Fi or Bluetooth). It must not use any intermediate cloud service or central server, in adherence to the local-only data storage policy (REQ-STO-01, REQ-STO-02).

=== Data Export

- *REQ-EXP-01*: The application shall provide a feature for users to export their data to a file.
- *REQ-EXP-02*: The export formats shall be JSON, XML and CSV, and the file shall be saved to a user-accessible location on the local device.
- *REQ-EXP-03*: The user must be able to export all data associated with a specific profile (e.g., all recorded health metrics).
- *REQ-EXP-04*: The user must be able to export application-wide configuration data, such as custom-defined exercises or equipment.

=== Data Import

- *REQ-IMP-01*: The application shall provide a feature to import data from a file, allowing users to restore or add data.
- *REQ-IMP-02*: The import functionality must support the same formats available for export (JSON, XML, CSV).
- *REQ-IMP-03*: It shall be possible to import data for:
    ** The currently active user profile (health metrics, etc.).
    ** The application-wide list of user-defined equipment.
    ** The application-wide list of user-defined exercises.
- *REQ-IMP-04*: During the import process, if an entry from the file has an ID that already exists in the local database, the system must prompt the user for action.
- *REQ-IMP-05*: The user prompt for handling duplicate entries must provide the following choices:
    ** **Ignore**: The entry from the file is skipped, and the existing data is kept.
    ** **Override**: The existing data is replaced with the entry from the file.
- *REQ-IMP-06*: The prompt shall include an option to apply the selected action (Ignore or Override) to all subsequent duplicates found within the same import operation.
- *REQ-IMP-07*: The user shall be able to import a complete user profile from a file.
- *REQ-IMP-08*: When importing a user profile, the user must be presented with the following options:
    ** **Create a new profile**: A new profile is created with the data from the file.
    ** **Merge with an existing profile**: The user can select one of the existing profiles to merge the imported data with.
- *REQ-IMP-09*: If "Create a new profile" is selected and a profile with the same ID already exists, the user must be prompted to either:
    ** **Cancel** the import for this profile.
    ** **Create a new profile with a different ID**: A new profile is created with a new unique ID, but with the data from the imported file.
- *REQ-IMP-10*: If "Merge with an existing profile" is selected, the duplicate handling mechanism from *REQ-IMP-04*, *REQ-IMP-05* and *REQ-IMP-06* applies to the data within the profile.
